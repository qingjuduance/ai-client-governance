# 文档维护要求

## AI 规则仓库会话同步

- 如果当前项目安装了本规则仓库，开启新的 Codex 会话后，必须先运行项目根目录下的
  `check-ai-rules-sync.ps1`，或等价调用规则仓库中的同名脚本。
- 同步检查不是系统定时任务；它只在新会话启动或用户明确要求检查时触发。
- 检查脚本读取 `.codex/ai-rules-config.json` 或规则仓库本地状态文件，
  只有距离上次成功同步或推送超过 24 小时时才执行一次 `sync-ai-rules.ps1`。
- 同步脚本可以提交规则仓库本地变更、拉取远端、合并并推送；发生合并冲突时必须
  停止并保留现场，不得自动覆盖任一方规则。
- 子 AI 是否启动由任务量评估、任务树、写范围耦合度和验证风险决定；不应额外要求
  用户重复点名“启用子 AI”后才执行本仓库已要求的多智能体分工。

## 协作审批流程

- 本仓库任务默认按 Plan mode 的协作节奏执行：无论当前界面或上层配置是否显示 Plan mode，开始修改文件、移动文件、运行会改变仓库状态的脚本、批量处理或导出前，都必须先在对话中给出带编号的处理计划、预计修改文件、验证方式和风险边界，等待用户明确按标签批准后再执行。
- 只读定位、读取 README/AGENTS、复用会话缓存、创建或更新 `.codex/task-tracking/` 任务跟踪文档，属于形成计划和审计链路所需的准备动作；准备动作完成后，仍要在正式修改或执行前向用户请求批准。
- 形成计划前必须先评估任务量：预计修改文件数、具体对象数、目录跨度、
  是否需要联网/源码核对、引用记录同步、pending/corrections 同步、
  Git 收口或多轮验证。只要达到大任务触发条件，就必须把计划写成
  智能体组和任务树方案；小任务不采用智能体组时，也要在 task tracking
  或当前回复中说明判断依据。
- 请求用户批准时，计划必须使用稳定且语义明确的标签，例如
  `计划-提交恢复机制`、`计划-迁移Java尾项`、`计划-更新引用记录`；
  禁止使用 `计划A`、`计划B`、`计划C` 这类只有序号、看不出任务范围的模糊名称。
  每个标签都要对应清晰范围，不能在后续补充需求时让旧计划自动失效。
- 请求用户批准时，最后必须单独写一行 Markdown 三级标题，格式优先为
  `### 请回复：批准：计划-提交恢复机制 / 批准：计划-迁移Java尾项 / 批准：全部`。
  不要把“批准”放进反引号、代码块或灰色代码样式里，确保它在 Codex 界面中足够醒目。
- 如果只有一个计划，也必须使用能说明范围的批准标签，例如
  `批准：计划-更新恢复现场`；用户追加新要求时，先判断它是
  补充旧计划、替换旧计划，还是新增计划，未确认前不得忽略此前计划。
- 用户回复 `批准：全部` 时，只批准当前这条请求批准消息中列出的全部计划；
  不自动批准历史消息中已过期、被替换或未重新列出的计划。
- 如果用户只回复普通“批准”，且当前请求批准消息只有一个计划，可以按
  该计划的具体标签执行，并在 task tracking 记录判断依据；如果有多个计划，
  必须先追问用户批准哪个计划或是否全部批准。
- 如果系统或开发者指令限制无法真正切换到 Plan mode、无法调用 Plan mode 工具，仍要用普通对话模拟 Plan 流程：先给带标签的计划并请求批准，用户回复对应批准标签后再实施。

## 文档架构优先级

- 目录组织是本仓库文档体系的架构。后续不管是新增知识点、重写路线、
  整理项目导读、迁移旧文件还是生成 README，都必须先判断目标目录、
  文件职责、索引关系和后续扩展性，再写正文。
- 参考 Java 项目、DDD、分层架构或其它工程目录时，只参考职责边界、
  依赖方向、模块内聚和扩展思想，不能机械照搬 `domain`、`application`、
  `infrastructure`、`interfaces` 等源码包名作为文档目录名。
- 新正式学习材料优先进入 `docs/`。方向材料放
  `docs/tracks/<方向>/`，跨方向通用材料放 `docs/shared/`，迁移后不再作为
  主线的历史材料放 `docs/archive/`。
- 每个方向目录优先使用面向学习者和维护者的固定结构：
  `README.md`、`roadmap.md`、`knowledge/`、`projects/`、`runtime/`、
  `integrations/`、`questions/`。
- `knowledge/` 放语言、框架、中间件、数据库和核心机制；
  `projects/` 放项目顺线、业务流程、源码入口和调用链；
  `runtime/` 放构建、配置、启动、部署、日志和排查；
  `integrations/` 放 API、字段、Feign、MQ、第三方服务和外部交互边界；
  `questions/` 只放卡点沉淀，不作为正式阅读路线入口。
- `code/`、`quick-review/`、`interview/`、`delivery/` 不作为新的长期目标目录。
  旧内容迁移时，源码入口收编进 `projects/`、`runtime/` 或 `integrations/`；
  速记、面试口述和简历表达只作为对应主题文件的局部表达或归档材料。
- `basics/`、`frameworks/`、`tools/`、`learning/` 在迁移完成前只作为旧资料来源
  和历史兼容入口；新内容默认不要继续强化这些旧顶层分类。
- 厚知识模块优先拆成 `README.md`、`basic.md`、`advanced.md`、`project.md`：
  `basic.md` 讲基础概念和最小示例，`advanced.md` 讲机制边界和排查，
  `project.md` 讲当前项目源码入口、调用链、字段、配置和验证方式。
- 如果用户纠正目录命名、目录职责或文件组织方式，必须优先更新架构说明、
  迁移矩阵、task tracking 和必要的 corrections，再继续写具体知识正文。

## 学习问答沉淀

- 用户粘贴代码、Markdown 片段或学习笔记并要求解释、修改、续写、纠错、整理时，必须先使用本仓库 `.codex/skills/locate-pasted-content/` skill 的流程，用 `rg` 定位该片段来自当前仓库的哪个文件或章节，再决定回答和修改位置。
- 定位到原文文件或章节后，不能机械把详细解释写回原文件；如果原文件是总览、索引、README、速记卡、题库引用，或原章节粒度与用户要求的细分知识点冲突，必须在对应目录的 `questions/` 下创建或更新独立聚焦文件，原文件只保留概要，不反向链接 `questions/` 文件，避免文档定位冲突和循环引用。
- 总览、专题主文档、路线文档和速记卡不能把多个细分知识点塞成一大段；遇到 Spring 事务、MyBatis-Plus、Redis、XXL-Job、Maven、Linux 工具这类可拆分主题时，主文档只保留模块定位、核心用途、项目落点和少量代表代码，详细解释按“一个框架概念/一个代码链路/一个面试追问”拆到对应 `questions/` 子目录。
- 用户问到语法、语言基础、面试题回答、项目相关概念时，必须先在当前对话中直接展示解释内容，不能只写入文档后简单告知路径。
- 是否沉淀为文档要看当前问题关联的目录，只写入当前相关目录下的 `questions/`，不要默认同时写入 `basics/questions/`、`notes/questions/`、`learning/questions/`。
- 如果问题来自或主要关联 `basics/` 的语言基础文档，只沉淀到 `basics/questions/`；如果主要关联 `learning/` 的项目学习文档，只沉淀到 `learning/questions/`；如果主要关联 `notes/` 的补充说明或约定，只沉淀到 `notes/questions/`。
- 每个目录下的 `questions/` 不能长期只做平铺文件堆积；新增或重新整理问答时，必须先按学习方向、语言、框架、项目或工具域建立子目录，例如 `basics/questions/cpp/`、`learning/questions/cpp/chromium/`、`learning/questions/cpp/audio-video/`、`frameworks/questions/qt/widgets/`、`frameworks/questions/qt/cmake/`、`tools/questions/chrome-cdp/`。
- `questions/` 子目录层级可以有多层，第一层优先用学习方向或技术域，例如 `cpp/`、`java/`、`go/`、`rust/`、`qt/`、`cmake/`、`linux/`；第二层再按项目、模块或知识族拆分，例如 `chromium/`、`audio-video/`、`widgets/`、`model-view/`、`build-deploy/`。避免为单个零散问题创建过深空目录，通常控制在 2 到 3 层。
- 整理旧 `questions/` 平铺文件时，必须同步清理 README、总览、速记卡、题库和正文中指向旧 `questions/` 路径的链接；移动后要用 `rg` 检查旧路径没有残留。未清理旧链接前，不要只移动文件。
- 即使不同目录都建立了同名方向子目录，也不能因此跨目录重复沉淀；仍然先按 `basics/`、`learning/`、`frameworks/`、`tools/`、`notes/` 的内容职责判断唯一归属。
- 只有用户明确要求多入口同步，或同一知识点确实同时作为多个目录的正式入口时，才在多个 `questions/` 目录中创建同名文件；这种情况下同名文件内容必须保持一致。
- 文件名必须直接体现知识点，一眼能看出文件在介绍什么。
- 禁止使用 `当前对话`、`问题整理`、`语法详解` 这类泛名作为主文件名。
- 语言基础主文档只保留概要说明，不反向链接 `questions/` 文件；详细解析放入对应的 `questions/` 文件。
- 仓库内 Markdown 链接使用相对路径，不写本机绝对路径。

## 文档引用层级与防循环

- 正式主文档、目录 README、总览、速记卡、题库和正文入口不能引用 `questions/` 下的文件；只能引用真实源码、基础模块、框架材料、工具材料和官方文档。对应仓库目录优先是 `.source-projects/`、`basics/`、`frameworks/`、`tools/`。
- 正式主文档和 README 如需说明存在独立问答沉淀，只能用纯文本说明 `questions/` 目录，不写 Markdown 链接，不把 `questions/` 当作阅读顺序入口。
- `questions/` 知识点文件不能反向引用主文档、README、总览、速记卡、题库，也不要串联引用同目录或其它目录的 `questions/` 文件；它们应直接引用真实源码、官方文档，必要时只引用更基础的 `basics/`、`frameworks/`、`tools/` 材料。
- 禁止形成 `主文档 -> questions -> 主文档`、`README -> questions -> README`、`questions -> 主文档/README -> questions` 这类循环路径。发现互链时，优先保留源码、基础、框架和工具链接，删除 `questions/` 与外层入口之间的链接。

## 目录级引用记录与递归同步

- 每个维护 Markdown 文档的目录都使用隐藏目录 `.references/` 记录引用关系；
  `.references/` 只保存维护元数据，不作为正式学习材料、README 入口或阅读顺序入口。
- `.references/` 里的记录文件必须与外层被维护文件同名。例如
  `learning/.references/Java后端知识点与项目双路径学习路线.md`
  对应 `learning/Java后端知识点与项目双路径学习路线.md`。
- 每个 `.references/同名文件` 只记录外层同名文件内部发出的 Markdown 引用，
  至少包含源行号、链接文本、原始链接、目标文件、目标锚点、目标引用记录文件、
  当前状态和处理备注。跨目录引用也记录在“发出引用”的当前目录记录文件中。
- 新增、删除或修改仓库内 Markdown 链接后，必须同步更新当前目录
  `.references/同名文件`；如果当前目录还没有 `.references/`，先创建该隐藏目录。
- 重命名、移动、删除 Markdown 文件，或修改标题、标题层级和锚点前，必须先用
  `rg` 在各目录 `.references/` 中反查旧文件名、旧路径、旧标题文本和旧锚点；
  命中后先更新引用方正文，再更新引用方对应的 `.references/同名文件`。
- 引用变更要递归处理：如果本次更新了某个引用方文件，还要继续查看该引用方自己的
  `.references/同名文件`，并继续反查是否有其它目录引用它，直到旧路径、旧锚点和
  旧链接文本没有残留。
- 处理引用关系时必须检查循环引用，不能形成 `A -> B -> A`、
  `A -> B -> C -> A` 这类链路；主文档、README、总览、速记卡、题库和
  `questions/` 之间尤其要检查入口循环。
- 发现循环引用时，优先保留真实源码、基础模块、框架材料、工具材料和官方文档链接，
  删除或改写会造成循环的学习入口、README、速记卡、题库或 `questions/` 链接。
- 每次处理 Markdown 引用、目录结构、文件重命名、标题锚点或 `.references/` 记录时，
  都必须在本次任务跟踪文档中记录“循环引用检查”，至少写明检查范围、检查方式、
  是否发现循环引用和最终处理结果；即使未发现循环，也要明确写“未发现循环引用”。
- `.references/` 记录文件中的仓库路径统一使用仓库相对路径或相对于当前目录能看懂的
  相对路径，不写本机绝对路径；记录文件本身也必须使用 UTF-8 保存。

## 写作风格

- 回答用户问题、解释代码、生成知识点或写正式文档前，必须先执行“回答质量门禁”：
  判断问题类型是概念、源码、链路、字段、配置、面试口述还是文档生成；
  第一段先正面回答用户问题，再展开背景、机制、边界和项目例子。
- 简单问题也要避免含糊：至少讲清“是什么、为什么、怎么用、容易错哪”中
  与当前问题相关的部分；复杂知识点默认补基础概念、关键机制、易错边界、
  项目落点和可口述表达，不能只围绕用户点名的单个词打转。
- 项目相关回答必须优先说明源码入口、调用链、关键对象或字段、
  结果影响和排查方式；链路类回答还要覆盖入口、注册或注入位置、
  调用方、被调用方、触发条件、执行顺序、传递对象、关键字段、
  落库/缓存/响应/异常影响。
- 字段、参数、配置项、枚举值、状态值或策略 key 类问题，必须说明值从哪里来、
  在当前链路里是什么角色、和哪些字段一起参与判断、最后影响什么结果。
- 不确定内容必须明确标注：哪些由源码或官方文档确认，哪些只是基于可见代码推断，
  哪些因为缺少源码、配置或运行环境当前无法确认；不能把推断写成确定结论。
- 如果回答会沉淀为文档，先在当前对话给出可读答案，再按目录归属、
  引用层级、联网核对、调用点记录和 task tracking 要求处理文档。
- 每个知识点文件优先包含：概念解释、常见代码示例、易错点、和当前项目或学习文档的关联说明；是否补充口述回答要看文档用途，不作为每题固定小节。
- 每次新增、补充、重写知识点、面试题答案、工具命令说明或专题文档时，都必须先联网搜索核对；即使内容看起来很熟，也要优先查官方文档、规范、源码、权威手册或可靠资料后再写。联网不可用或找不到权威来源时，要在任务跟踪文档和最终回复中说明原因，并避免把未核对内容写成确定结论。
- 写知识点和面试题答案必须准确第一、内容饱满但不灌水：能一句话讲准的就一句话讲完；需要展开的再补机制、示例、边界、参数含义、项目关联或排查方式。不能为了凑格式扩写，也不能因为想压缩而省掉关键机制、易错点或必要例子。
- Markdown 正文以源码阅读体验为准，普通说明行目标控制在约 80 个可见字符，
  尽量不超过 100；超过 120 时必须按语义拆成短段、短列表、引用式链接、
  多行代码块或更窄的表格结构，避免编辑器横向拖动。
  中文正文可把 40 到 60 个汉字加少量英文、代码视为一行的舒适范围。
  长 URL、代码、命令和确需保留的表格可以例外；
  但如果已经影响阅读，应优先改成列表、分层结构或引用式链接。
- 面试题、速记卡、专题文档和 `questions/` 知识点不要用一个泛泛的 `要点：` 把定义、机制、示例、边界和项目落点混在一起，也不要机械套用固定小标题。简单知识点优先用 1 到 3 个短段落或短列表直接讲清楚；复杂知识点才使用少量语义小节，通常控制在 2 到 3 个，例如 `回答`、`机制`、`易错点`，确有项目关联时再补 `项目相关`。
- 成篇整理题库或专题文档时，不能只修用户举例的单个问题；必须逐题或逐节判断答案是否准确、是否需要补充、是否应该压缩，以及是否需要拆成短答、列表、示例或分层说明。
- 面试题答案第一句必须先正面回答题目本身，再展开机制和边界；不能把随机实现细节、参数名或项目经验放在定义位置。例如 Java 对象创建过程要先讲类加载检查、内存分配、零值初始化、对象头设置和构造器执行，再把 TLAB、指针碰撞、空闲列表、对象头 Mark Word 归入 HotSpot 实现细节。
- 涉及 JVM、Java 语法、标准库、框架、数据库、中间件和工程命令时，修改或新写正式文档前要在联网核对基础上区分“语言/规范层面的语义”和“某个实现的细节”。例如 `new`、字段默认值和构造器调用属于 Java 语言/JVM 规范语义；TLAB、对象头布局、指针碰撞属于 HotSpot 等 JVM 实现细节。
- 工具命令、脚本命令、工程化命令，以及数据库排查字段、框架/中间件配置字段、
  枚举值、状态值和命令输出字段，必须说明作用、关键取值含义、占位符含义、
  常见使用场景、判断方式、风险边界和验证方式。例如 `ps aux` 不能只写
  “查看进程”，必须解释 `a`、`u`、`x` 分别代表什么；`docker logs -f api`
  必须解释 `-f` 是持续跟随日志，`api` 是容器名或容器 ID，不是镜像名；
  `EXPLAIN.type` 不能只写“访问类型”，必须解释 `const/ref/range/index/ALL`
  等常见取值和它们要如何结合 `key`、`rows`、`Extra` 判断。
- 每次新增、补充或重写知识点文件时，必须先补充基本知识，再结合当前项目进行知识补充和介绍。基本知识部分要先讲通用概念、核心术语、适用场景、最小代码示例和官方依据；项目结合部分再讲当前项目的源码入口、依赖配置、业务链路、实际使用边界、排查点和可口述经验，不能只堆项目源码，也不能只写抽象概念。
- 生成、补充或重写框架、语言基础、中间件、数据库、工程工具、
  项目链路等知识点时，默认按“初级、进阶、实战”三层组织内容。
  该要求适用于 Java 简历驱动知识点，也适用于后续其它方向知识点：
  - 初级：面向初学者讲清基础概念、核心术语、适用场景、最小示例、
    版本或标准来源和官方依据，让第一次接触的人能先看懂是什么、为什么用。
  - 进阶：补高级用法、关键机制、生命周期、配置项、常见组合、
    边界条件、易错点、性能或一致性影响，以及排查和验证方式。
  - 实战：结合当前简历、项目源码或学习项目说明源码入口、依赖配置、
    业务链路、字段和配置取值、日志/数据库/接口验证方式、风险边界和面试表达。
- 简单卡点问答可以不机械写三个大标题，但仍要覆盖“基础是什么、
  进阶边界是什么、项目怎么落地”三个问题；复杂专题、正式系统知识文档
  或简历驱动知识清单，应使用显式三层小节、表格或同等清晰结构。
  不得只列简历术语清单，也不得只堆项目源码当作知识点解释。
- 新增、重写或大幅补充任何知识点、面试题答案、专题文档或项目导读前，
  必须先做“新知识点覆盖检查”，不能只围绕用户点名的单个术语写内容。
  覆盖检查至少包含：上位概念、同层依赖、下位机制和项目落点。
- 上位概念要检查该知识点所属语言、标准、框架、规范、中间件或一级模块；
  同层依赖要检查常一起出现的组件、配置、协议、工程工具和运行环境；
  下位机制要检查关键注解、接口、配置项、生命周期、执行链路和数据结构；
  项目落点要检查当前项目源码入口、依赖配置、业务链路、排查方式和面试表达。
- 以 Java 后端为例，新增 Spring Cloud/OpenFeign、Spring MVC、事务、
  MyBatis-Plus、Redis、XXL-Job、RocketMQ 或 Maven 等知识点时，
  必须主动检查是否缺少 Spring Boot、Spring Framework、Servlet/Web、
  AOP、IoC、配置绑定、自动配置、starter、依赖管理、启动链路、数据库、
  缓存、消息队列和项目运行排查等上位或相邻知识入口。
- 覆盖检查发现关联知识缺少正式入口时，不要硬塞进当前文件；
  应按 `basics/`、`frameworks/`、`tools/`、`learning/`、`interview/`
  的职责判断唯一归属，必要时列入本次计划、同步补充或写入未处理项。
- 如果本次只处理覆盖矩阵中的某一个点，必须在 task tracking 写清
  “已检查的关联知识”“本次补充的范围”“暂不补的关联点和原因”；
  不能等用户逐个提醒后才补 Spring Boot、SpringCloud、框架基础或其它相邻点。
- 新增、重写或大幅补充知识点、专题文档或项目导读后，必须做
  “学习路线同步检查”：回看相关学习路线、目录 README、速记卡、题库、
  项目导读和面试入口是否需要同步入口、阶段顺序、知识点名称、项目落点
  或未处理项；如果只是新增卡点问答且不改变正式学习顺序，也要在
  task tracking 写明不更新路线文档的原因。
- 解释框架继承、泛型基类、自动注入、模板类、代理类或封装工具时，不能只贴“extends/implements”代码；必须说明每个泛型参数代表什么、父接口/父类提供了哪些能力、关键方法从哪里继承或注入而来，以及当前项目为什么选择这种写法。例如 MyBatis-Plus 的 `ServiceImpl<Mapper, Entity>` 必须讲清 `Mapper extends BaseMapper<Entity>`、`Service extends IService<Entity>`、`ServiceImpl` 注入 `baseMapper` 并实现通用 CRUD。
- 解释 MQ、RPC、HTTP 客户端、数据库访问、缓存客户端、公司 starter
  或其它封装层时，必须先讲官方或框架原始接口，再讲项目封装接口如何映射；
  同时区分业务对象、框架包装对象和真实传输或执行格式，说明哪些字段进入
  body、header、key、tag、properties、SQL 参数或协议元数据。
- 解释任何字段、参数、配置项、判断数据、枚举值、状态值、策略 key、
  白名单、缓存 key、消息字段、SQL 条件、请求头或请求体字段时，不能只说
  “用于判断”“用于传递”或“用于配置”；必须说明该值在当前链路中的角色：
  是计算维度、策略类型、具体业务值、白名单集合、运行时请求上下文、
  传输字段、落库字段、缓存键组成、判断阈值还是结果状态；同时说明值从哪里来、
  和哪些字段一起参与判断、最后影响什么结果。
- 解释或写作框架、认证、过滤器、拦截器、AOP、RPC、MQ、数据库、缓存、
  客户端/服务端等链路时，必须记录关键调用点，不能只写抽象流程图、
  不能只列源码入口。调用点至少包括入口方法或配置、注册或注入位置、
  调用方、被调用方、触发条件、执行顺序、传递对象、关键字段、
  落库/缓存/响应/异常影响，以及对应源码文件和行号或文档标题锚点。
  当前对话回答、正式文档和 task tracking 缓存索引都要保留这些调用点，
  方便后续追问、复盘和继续写文档时直接复用。
- 内部 starter、公司包或封装工具源码缺失时，不能只写“可能封装了一层”；
  先讲官方文档、源码或可见接口能确认的机制，再明确项目封装无法确认的边界，
  避免把不确定性写成读者无法落地的空泛推断。
- 当框架、中间件或基础设施知识点文档持续被项目源码细节挤占时，
  必须拆分为“系统知识入口”和“项目落地说明”：系统入口讲通用机制、
  核心术语、边界、易错点和官方依据；项目落地说明讲源码入口、业务链路、
  配置、排查点和可口述经验。
- 生成或重写框架、中间件、分布式系统、数据库、工程工具等文档时，
  必须先判断是否需要同时产出两个职责不同的文档或入口：
  “项目顺线文档”和“系统知识文档”。项目顺线文档顺着当前项目源码、
  `pom.xml`/构建文件、启动类、配置类、Controller/Service/Mapper/Feign、
  `src/main/resources` 下的 `bootstrap*`、`application*`、mapper XML、
  logback、模板和运行配置讲清项目怎么用、怎么启动、怎么调用、怎么排查；
  系统知识文档脱离当前项目，按官方文档、规范、源码或权威资料讲清概念、
  架构、组件职责、核心流程、边界、易错点和面试表达。
- 不能把系统知识和项目落地硬塞成一篇大杂烩。若项目顺线和系统知识都需要，
  应拆成两个文件、两个清晰小节或两个明确入口；正式主文档、README、总览、
  速记卡和题库仍不得把 `questions/` 当正式阅读顺序入口。
- 项目顺线文档必须随本地源码快照变化重新核对。只要 `.source-projects/`
  或 `src/main/resources` 被恢复、替换或更新，就要重新检查旧文档中
  “当前快照缺少配置”“无法确认地址、DataId、namespace、group、profile”
  等保守判断是否过期。
- 系统知识文档必须联网核对，优先官方文档、规范、源码、权威手册和项目官网；
  可以参考优秀教程的组织方式，但正式结论必须以官方或可验证来源为准。
  task tracking 中要记录联网来源、采用了哪些结构、哪些资料只作参考未采用。
- 读取或引用 `resources`、配置文件、证书、密钥、token、账号密码、内网地址
  和第三方访问凭据时，正式文档只写配置键、职责、脱敏示例和排查方式，
  不直接复制真实敏感值；如需说明存在敏感项，只描述类别和风险边界。
- 新增、重写或大幅补充 Markdown 文件时，必须在标题后尽早补一个轻量索引区，例如“快速索引”“文档索引”或对应目录约定的索引表，便于后续用 `rg` 和按段读取。索引区应短小，优先包含：关键词、适用范围、核心问题、项目源码入口、官方依据、当前文件主要小节；不要把正文解释复制到索引里。
- 整理或修改已有 Markdown 文件时，要顺手检查该文件是否缺少索引、索引过旧、标题层级混乱或多个主题混在一节；如果本次修改触及该文件的结构或入口，就同步做局部规范化和索引化。不要为了索引化无关文件而临时发起全仓库大改；用户明确要求全量规范化时，才按批量任务建立清单逐项处理。
- 索引化不能破坏文档引用层级：主文档、README、总览、速记卡和题库不能在索引里链接 `questions/` 文件；`questions/` 文件的索引也不能反向链接主文档、README、总览、速记卡或其它 `questions/` 文件。索引中的 Markdown 链接仍按允许层级指向真实源码、官方文档、基础模块、框架材料或工具材料；不适合链接的只写纯文本关键词。
- 需要口述回答时，表达要简洁、可直接说出口，不要写成过强经验；但不要把 `面试表达` 固定成每个题目的必备小标题。
- 示例代码要短，突出语法和使用场景。
- 涉及语言语法、标准库、核心 API 或框架库时，必须说明它大概是哪个版本/标准引入的，例如 C++11、C++17、Java 8、Go 1.13；如果只是早期已有或无法精确到小版本，要写清楚“早期标准库已有”或“常见于某版本之后”。
- 涉及语言语法、标准库、核心 API、框架库、构建工具或重要配置项时，回答和正式知识点文档必须附带官方文档链接；链接要优先定位到官网中对应类、函数、属性、信号、命令、参数或章节锚点，不能只给官网首页或泛目录。如果官方没有更细锚点，要说明已定位到最近的官方章节。
- 解释网络连接、代理隧道、握手流程、链路建立、客户端/中间件/服务端交互等多端通信场景时，优先补充 `text` 链路图或 Mermaid 时序图，标清谁和谁建立 TCP/UDP 连接、TLS/HTTP 发生在哪一层、代理/网关是转发字节流还是终止协议，方便区分直连、代理转发和加密会话关系。
- 框架链路图、时序图、流程图或 Mermaid 图用于解释当前项目时，
  图后必须紧邻补充项目源码对照，至少说明依赖入口、启动类、业务配置、
  自动配置边界和显式手写代码；如果某一步由 starter 自动完成，要标清是
  基于依赖和配置推断，而不是项目手写调用。
- 仓库维护的 Markdown、配置和脚本文件必须使用 UTF-8 编码保存；读取、批量替换、脚本校验时也要显式按 UTF-8 处理，避免 Windows 默认 GBK 导致中文路径或中文内容乱码。
- `.source-projects/` 中的外部源码快照原则上保留上游原始编码，不为满足仓库文档编码规则强行转码；本仓库新增的说明文件和学习示例项目仍必须使用 UTF-8。
- Markdown 正文中一旦引用仓库内其他具体文件，必须写成可点击的相对 Markdown 链接，不能只用反引号包路径；只要引用的是另一个文件里的某个具体位置，而不是把目标当作整个文件入口，就必须落到对应标题锚点，并在链接文本里写清文件名和小节名，不能只跳到文件顶部；只有引用整个文件入口或目录总览时，才允许只链接到文件本身；目录引用不需要加跳转链接，可以继续用反引号标识；命令、参数、代码标识符和不适合作为链接的纯代码片段继续使用反引号。
- 面试高频问题不能只列问题清单，必须补充可直接口述的答案；如果项目里有对应例子，必须在答案旁边补相对 Markdown 跳转链接，优先链接到 `code/` 关键代码导读，确实存在源码快照时再链接到 `.source-projects/` 具体文件。

## 文档规则合规检查

- 每次处理已有 Markdown 文档或刚创建的 Markdown 文档后，都必须把该文件作为
  独立对象检查：它属于哪个目录、哪类文档，是否已符合当前 `AGENTS.md`、
  所在目录 `README.md`、目录层级约束、写作风格、索引、引用层级、
  `.references/`、UTF-8、联网核对、脚本验证、简历导出等适用维护规则。
- 检查不能只看本次新增段落。已有文档被修改后，要顺手检查本文件已有结构
  是否存在明显不符合当前规则的问题，尤其是缺少或过旧索引、章节混杂、
  链接层级错误、`questions/` 互链、绝对路径、长行、泛名标题、README 或
  `.references/` 未同步等；如果问题在本次触及范围内或会影响本次产出，
  必须一起修正。
- 如果发现的旧问题明显超出本次需求范围，不能静默忽略；必须记录到本次
  task tracking 的“未处理文件”“检查不通过或需要继续处理的文件”或
  “风险边界”中，并在最终回复说明剩余项和不处理原因。
- 新建文档在交付前必须完成基础合规检查：文件名见名知意、标题后有轻量索引、
  内容归属唯一、仓库内链接使用相对路径和正确锚点、不把 `questions/` 作为
  正式入口、UTF-8 保存，并按规则同步 README、`.references/` 或导出结果。
- 本次处理的每个正式文档、README、题库、速记卡、简历、`questions/`、
  corrections、task tracking 或 skill 文件，都要在本次 task tracking 的
  “已处理文件”或“处理后已检查且满意的文件”中写清合规检查结论；
  不能只写“已修改”。

## 会话文件缓存与搜索优先级

- 当前不是刚刚启动的会话时，必须优先复用本会话已经读取、定位、解释或修改过的文件列表，把这些文件视为当前任务的上下文缓存。
- 新建本次任务跟踪文档不能替代旧缓存复用；但新会话或用户说“继续未完成任务”
  时，默认先读取 `.codex/pending-tasks/index.md` 的当前活跃任务和命中的单个
  pending 文件，只按该 pending 的最小恢复读取清单继续。只有当前活跃 pending
  明确要求回看上一任务链、最小恢复清单不足以判断状态、或用户连续追问同一文件/
  章节/项目时，才沿 task tracking 的“会话链路”逐级回溯；回溯时只抽取
  `会话链路`、`缓存索引`、`已处理文件`、`最终结论` 等命中段，不追溯无关历史任务。
- 每份任务跟踪文档必须形成会话链：记录“上一任务跟踪文档”“本次复用的旧缓存文件”“本次新增缓存文件”。如果是本会话第一份任务跟踪文档，要明确写“上一任务跟踪文档：无，本会话链头”。如果无法确认上一份是否属于当前会话，不使用时间推断，直接记录“上一任务跟踪文档：无法确认，本次作为新的链头”。
- 每份任务跟踪文档必须在“会话链路”之后维护一个精简“缓存索引”小节，用短行记录本次涉及文件，格式优先为 `文件路径 | 关键词/主题 | 用途 | 关键结论`。缓存索引用于后续沿链快速判断是否需要读取该文件，不能写成长篇解释。
- 沿会话链复用旧缓存时，不默认全文读取每个 tracking Markdown；必须先用 `rg`、标题定位或小范围读取，只抽取 `会话链路`、`缓存索引`、`已处理文件`、`最终结论` 等短段。只有当缓存索引命中当前问题、需要核对具体结论，或要修改对应文件时，才读取相关 tracking 或正式文档的全文。
- 如果未来反复需要执行同一套链式缓存抽取流程，可以把“沿链抽取缓存索引”的命令和判断规则沉淀为本仓库本地 skill 或脚本；但即使使用 skill，也只读取索引段和命中段，不把整条链的 Markdown 全量读入上下文。
- 如果用户连续追问同一文件、同一章节、同一项目或同一知识点，优先读取上一任务跟踪文档，并沿会话链向上查找旧“已查看文件缓存”和“已处理文件”；不能只新建一个跟踪文档后直接全仓库搜索。
- 每次读取、定位、解释或修改文件后，都要在本次 `.codex/task-tracking/` 跟踪文档中记录到“已查看文件缓存”或“已处理文件”里，至少包含文件路径、查看目的和关键结论，方便后续问题优先复用。
- 用户继续追问同一项目、同一章节、同一代码片段或同一目录时，先在已查看文件缓存中用 `rg`、精确片段或函数名搜索；如果缓存文件不能解释当前问题，再扩大到当前目录、相邻文件、最近 README 或全仓库。
- 对粘贴代码、Markdown 片段或学习笔记进行定位时，也要先检查已查看文件缓存；如果未命中，再按 `locate-pasted-content` skill 的流程搜索仓库。
- 已查看文件缓存只作为搜索优先级和上下文复用依据，不能替代实际文件内容核对；回答、修改或引用前仍需确认当前文件内容没有变化。
- 如果用户切换到明显无关的新主题、新目录或新项目，可以降低旧缓存优先级，但仍要把新读取的文件继续追加到当前任务跟踪文档。

## 检索与工具效果统计

- 每次使用会话缓存、文档定位、`locate-pasted-content` 或其它本地 skill 后，
  必须在本次 task tracking 中维护“检索与工具效果统计”小节。
- 统计不能只写“已使用 skill”；至少记录是否复用上一任务缓存、缓存命中文件数、
  新增读取文件数、`rg` 查询次数、关键查询词、命中范围、候选文件数、
  最终命中文件和是否需要扩大搜索范围。
- 每次读取文件后，要尽量记录实际读取文件数、读取行号范围或片段数量；
  如果通过缓存索引、小范围读取或精确 `rg` 避免了全仓库搜索、大文件全文读取，
  也要写清避免了什么范围。
- 使用脚本、导出工具、批量检查工具或其它会影响仓库状态、统计结果或检索结论的
  工具时，必须记录工具名称、输入范围、输出结论、成功或失败状态和验证方式。
- 涉及 token 消耗或节省时，必须写清数据来源：真实工具统计、模型上下文估算、
  文件/行数等代理指标，或“暂无数据”。没有真实 token 统计时，
  不得写“已减少 token”“节省了 token”这类确定结论，只能说明
  “通过少读文件、只读索引段等方式降低上下文输入量”，并列出代理指标。
- 如果连续多次同类任务都要手动统计检索效果或工具效果，先提出只读统计脚本计划，
  等用户批准后再实现。第一阶段脚本只输出报告，不自动修改 `AGENTS.md`、README、
  skill、corrections 或正式文档。

## 开发任务现成方案优先检索

- 做功能开发、Mod、插件、脚本、自动化、CLI、数据处理、项目改造或其它工具类任务前，
  必须先评估是否已有可复用的现成工具、开源库、Mod、插件、模板、命令行工具、
  官方示例或成熟方案；不能默认从零实现。
- 检索优先级是先联网搜索，再结合本地上下文核对。联网搜索优先看官方文档、
  项目官网、源码仓库、发行页、插件/Mod 平台、权威讨论和维护状态；本地核对要看
  当前仓库、`.source-projects/`、下载目录、已有依赖、README、脚本目录、
  运行时插件目录、游戏或工程的实际安装目录。
- 形成执行计划时必须写出“现成方案检索结果”：查了哪些关键词或来源、
  找到了哪些候选工具，最终是采用、改造、只参考还是放弃，并说明原因。
- 如果现成方案维护状态好、许可或使用边界清楚、能满足需求且风险低，
  优先集成、配置或小范围改造；只有在现成方案缺失、失效、风险过高、
  与目标强冲突或验证不通过时，才从零实现或自写脚本。
- 对 Mod、游戏运行时、逆向数据、浏览器自动化、构建发布、文档转换、
  文件批处理和图片/表格/文档生成等任务，尤其要先查是否已有工具链或同类 Mod；
  若已有方案只覆盖部分需求，应优先复用其数据来源、接口、配置格式或验证方法。
- 检索不到可靠方案、网页被拦截、资料过旧、版本不匹配或许可证不清时，
  必须在 task tracking 和最终回复中说明不可确认边界；不能把推断当成确定结论，
  也不能为了赶进度跳过现成方案搜索记录。

## 会话操作账本与恢复现场

- 每次处理用户提出的具体要求时，必须先创建或更新一个仅供处理过程使用的
  Markdown 跟踪文档，用来记录本次任务的处理范围、文件状态和检查结果。
- 跟踪文档统一放在 `.codex/task-tracking/`，文件名必须包含日期和任务关键词，例如 `.codex/task-tracking/2026-05-28-英文翻译检查.md`；该目录只记录处理过程，不作为正式学习材料入口。
- 每份 task tracking 都是当前会话操作账本，必须让后续新会话在不依赖聊天记忆的
  情况下判断“已经做了什么、做到哪一步、还能不能继续”。它不只服务文档迁移；
  解释、审计、代码修改、文档重构、导出、提交、验证、批量处理和工具规划都适用。
- 跟踪文档至少包含：本次需求概述、计划处理的文件清单、已查看文件缓存、
  已处理文件、未处理文件、无需处理文件、处理后已检查且满意的文件、
  检查不通过或需要继续处理的文件、最终结论。
- 跟踪文档还必须维护“操作账本”和“恢复现场”信息：
  原始用户请求、最新用户指令、批准标签、已执行操作、未执行操作、
  读取文件和行号范围、修改文件、运行命令和结果、验证命令、最近安全停止点、
  下一步动作、当前 Git 状态、最近 commit、子 AI 运行状态、禁止误动范围、
  最小恢复读取清单和可跳过的重复阅读范围。
- 对批量任务必须边处理边更新跟踪文档，例如“把文件里的英文翻译成中文”时，要逐项记录哪些文件已经翻译、哪些文件还没翻译、哪些文件翻译后已经复查并确认满意。
- 长任务、架构调整、批量迁移、批量生成、多 AI 协作、跨目录修改和任何可能被
  断电、死机、上下文压缩、用户暂停或新会话接管打断的任务，必须在每个稳定阶段
  更新一次恢复现场；稳定阶段包括完成一批文件、回收子 AI、跑完验证、准备 commit、
  commit 完成、发现阻塞或准备最终回复。
- 如果本次任务涉及 Markdown 引用、目录结构、文件重命名、标题锚点或 `.references/`
  记录，跟踪文档必须单独记录“循环引用检查”，包含检查范围、检查方式、
  是否发现循环引用和处理结论。
- 最终回复用户前，必须再核对一次跟踪文档，确认没有遗漏计划范围内的文件；如果有文件无法处理、暂未处理或检查不满意，必须在最终回复中明确说明原因和剩余项。
- 最终回复必须单独说明“剩余处理”，并明确范围：先说明本次批准计划范围内
  是否还有未完成项、未验证项、仍在运行的智能体组和已知风险；再说明
  会话级或仓库级是否还有 pending 活跃任务、未提交工作区、未 push 的明确需求、
  待提炼 corrections、后续审批计划或其它收口事项。
- 写“剩余处理：无”前，必须核对本次跟踪文档、`.codex/pending-tasks/`
  当前活跃项、智能体组状态看板、Git 工作区状态和 correction 状态；只有
  本计划范围与会话级/仓库级都没有待处理、未验证、未提交或需另开审批事项时，
  才能使用全局“无”。如果只是本计划闭环，必须写成“本计划范围内无剩余；
  仓库级后续：……”，不能把局部完成误报为全局无剩余。
- 跟踪文档是任务执行过程的自检依据，不能替代正式文档、README、问题沉淀文档或简历导出检查；如果任务本身要求产出正式文档，仍然按对应目录规则另外维护正式文档。

## 通用会话恢复队列

- `.codex/pending-tasks/` 用来记录未完成、暂停、阻塞、待批准、待恢复或
  可能因上下文丢失而无法继续判断的会话任务；它只保存过程恢复入口，
  不作为正式学习材料入口，也不替代 `.codex/task-tracking/`。
- `.codex/task-tracking/` 继续记录完整处理过程；`.codex/pending-tasks/`
  只保存“后续继续做时必须先看的恢复入口”。同一任务如果未完成，
  两边都要记录：task tracking 记录过程，pending 文件记录下一步恢复方式。
- 当用户暂停任务、临时离开、要求稍后继续、长任务中断、存在未批准计划、
  已批准但尚未执行完成、验证未通过、还有未处理文件，最终回复前发现计划范围
  没有全部闭环，或任务规模大到新会话重新阅读会明显浪费 token 时，
  必须创建或更新 `.codex/pending-tasks/` 中的任务文件。
- 对于还未处理完成的任务，pending 文件必须详细记录，至少包含：
  任务 ID、当前状态、关联 task tracking、批准标签、已批准范围、最近一次有效计划、
  原始用户请求、最新用户指令、已完成步骤、未执行步骤、涉及文件、运行命令、
  验证方式、风险边界、最近安全停止点、下一步动作、阻塞原因、当前 Git 状态、
  最近 commit、是否有子 AI 仍在运行、恢复时优先读取的文件、最小恢复读取清单、
  可跳过的重复阅读范围和禁止误动的文件。
- `.codex/pending-tasks/index.md` 是恢复队列索引，必须列出任务 ID、标题、
  状态、优先级、批准状态、关联 task tracking、最近更新日期和下一步动作。
  新增、完成、暂停或重新激活 pending 任务时，都要同步更新索引。
- 后续用户说“继续上次”“处理未完成任务”“恢复之前的任务”“继续迁移”
  或新会话上下文不足以判断未完成事项时，必须先读取
  `.codex/pending-tasks/index.md` 中的当前活跃任务区，再读取命中的单个 pending
  文件和其中列出的最小恢复读取清单；不能只靠当前对话或旧记忆猜测，也不能默认
  全仓重扫。已完成任务区只作审计，不作为默认恢复入口。
- pending 任务完成后，不删除任务文件；把索引和任务文件状态更新为 `已完成`，
  写清完成时间、最终结论、验证结果和是否仍有剩余风险，方便后续审计。

## Git 操作要求

- 用户只要求“提交”“commit”“提交一下”或类似本地提交操作时，
  默认只执行本地 `git commit`，不得执行 `git push`。
- 只有用户明确说“push”“推送”“提交并推送”或同等含义时，才允许执行
  `git push`；如果表述含糊，先询问并等待明确批准。
- 执行 `git add`、`git commit`、`git push` 前仍然遵守协作审批流程：
  先说明计划、预计 staged 文件、验证方式和风险边界，等待用户回复“批准”。
- 工作区存在无关脏改动时，只 stage 本次任务相关文件；不要把历史遗留修改、
  用户未要求的文件或不确定来源的改动带进同一个 commit。
- commit 后要检查提交号、最新提交信息和剩余工作区状态；除非用户明确要求，
  不要为了清理提示顺手执行 `git gc`、`git prune`、reset、checkout 等维护或
  破坏性命令。
- 长任务执行期间，每隔 1 小时必须进行一次 commit 检查，并在 task tracking
  记录检查时间、距上次 commit 或检查的间隔、当前是否有子 AI 仍在运行、
  是否存在未合并的子 AI 输出、准备 stage 的文件、验证结果、commit hash
  或跳过原因。
- 每小时 commit 检查不是无条件自动提交。执行 `git add` 和 `git commit`
  仍必须遵守协作审批流程；如果尚未获得批准、有子 AI 仍在运行、同一文件
  可能继续被改、验证未通过、存在无法分离的无关脏改动，或本次没有可安全
  提交的任务相关改动，则跳过 commit 并记录原因。

## 子 AI 协作负载与动态调参

- 每个执行任务在开始改文件、批量读取后准备正式处理、运行会改变仓库状态的脚本、
  启动导出或验证前，必须先做“任务量评估”。评估至少覆盖：预计修改文件数、
  具体知识点或模块数、是否跨目录或跨职责层、是否需要联网核对、源码核对、
  `.references/` 同步、pending/corrections 同步、Git 收口或多轮验证。
- 任务量评估结论必须写入本次 task tracking。小任务可以由主对话直接执行；
  但只要满足任一大任务触发条件，就必须先创建智能体组和任务树，再执行正文、
  代码、迁移或验证工作，不能事后补建。
- 大任务触发条件包括但不限于：写作、实现或资料整理对象超过 3 个；检查、审核、
  验证对象超过 5 个；任务跨 `docs/`、`.codex/`、`.references/`、源码快照或
  多个方向 track；需要同时处理正文、索引、引用记录、pending/corrections、
  状态看板和验证；用户明确要求继续完成一个未收口的大任务；或当前任务已经在
  pending 中登记为执行中、待验证、阻塞或可恢复任务。
- 如果任务量边界不清，先按大任务处理：建立智能体组、拆父节点和叶子节点、
  写清每个叶子的文件范围和验证方式。只有确认任务不超过阈值、无跨目录状态同步、
  无后台恢复现场时，主对话才可以独立执行，并在 task tracking 写明判断依据。
- 大型重构、批量文档生成、跨目录整理、批量代码修改或用户明确要求多 AI
  分工时，总控必须先拆出不重叠的文件范围和知识点范围，再创建子 AI。
- 子 AI 分工必须按“任务树”理解，不是扁平清单。父节点 AI 只负责拆分、
  判断阈值、创建必要子 AI、定义输入输出、整合结果、验收质量和记录状态；
  父节点不能直接承包具体正文、代码、迁移或验证任务。
- 只有叶子节点 AI 才能执行具体任务。叶子任务必须已经足够明确，至少写清
  输入来源、输出文件、禁止范围、验收方式和风险边界；如果某个子 AI 判断自己
  拿到的任务仍然超过阈值，它必须继续向下拆分子 AI，自己转为父节点。
- 创建任何子 AI 前，总控必须先设计上下文压缩输入包：优先生成
  `.codex/agent-briefs/` 下的 Agent Brief，或在 spawn 消息中给出等价短输入包。
  brief 至少写清任务范围、允许改的文件、必读文件、可跳过文件、已确认事实、
  禁止范围、验证命令、输出格式和 token 统计口径。子 AI 默认只读 brief、
  自己负责的文件、必要 `.references`、必要源码和官方链接；全局 AGENTS、
  pending、task tracking、corrections 和历史看板由总控读取后压缩传递，
  不让每个子 AI 重复全量阅读。
- 不默认使用完整历史上下文创建子 AI；只有任务确实依赖当前长对话细节、
  文件强耦合或用户明确要求时才使用，并在 task tracking 记录原因、风险、
  预计额外输入范围和可替代的 brief 方案。没有真实 token usage 时，不能声称
  “已节省 token”，只能按文件数、行数、字符数、brief 长度和是否传递完整历史等
  代理指标说明“降低上下文输入量”。
- 判断子 AI 是否具备某项能力前，总控必须先基于当前可见工具 schema、
  已启用插件和 skill 说明、现有脚本、状态看板和通信总线事实源做判断；
  只有这些来源无法确认时，才把“能力探测”作为明确叶子任务交给子 AI。
  探测任务必须写清只读/可写范围和回报字段，不能让子 AI 自行泛读历史上下文。
- 内置多智能体工具用于即时控制，例如创建、发送消息、等待、关闭和恢复子 AI；
  需要跨会话恢复、消息确认、心跳、交付物索引、token 数据来源或未读消息统计时，
  必须同步使用 `.codex/agent-comm/` 文件型通信总线。总线事实源由 UTF-8
  JSON/JSONL 文件和 `scripts/agent_comm.py` 管理；初版不做网络服务，
  不把 SQLite 或其它二进制数据库作为唯一恢复入口。
- 同时存在多个智能体组时，必须默认按并发场景处理：每组先登记 `group_id`、
  写入范围、通信组和恢复入口；同一文件或父子目录范围不得被多个组同时写入。
  如果确需并行处理同一文件，必须在 task tracking 中拆成不重叠标题、锚点或
  行段，并写明合并顺序、冲突策略和验证方式。
- 多组并发写入前必须使用 `.codex/agent-comm/` 的锁/租约记录写范围：
  `lock acquire` 获取锁，`lock release` 释放锁，`lock status` 查看全局锁。
  锁记录至少包含 group、owner、scope、lease_expires_at 和状态；阶段收口时
  `validate` 必须检查未 ack、重复消息、过期 heartbeat、锁冲突和租约状态。
- 子 AI 的最终输出和通信总线记录必须包含 `token_usage_source`：
  `real_tool_usage`、`unavailable` 或 `proxy_estimate`。只有工具明确返回真实
  usage 时才记录真实 token；没有真实数据时，只记录 brief 行数、必读文件数、
  预计读取行数、是否 full-history fork、可跳过文件数等代理指标。
- 默认阈值：写作型、实现型或资料整理型叶子 AI 最多负责 3 个直接具体任务
  对象；对象可以是 3 个知识点、3 个明确文件或 3 个清晰模块。这里的 `3`
  是“每个 AI 节点的直接子任务上限”，不是整棵任务树的全局总数。
- 检查、审核、整合型叶子 AI 默认最多负责 5 个直接检查对象；对象可以是文件、
  子目录、知识点组、子 AI 交付摘要或待合并变更。超过 5 个时必须继续拆分
  检查任务树或创建新的检查/整合 AI。
- 上述 `3` 和 `5` 是当前默认阈值，不是永久常量。每次使用多 AI 分工时，
  task tracking 必须记录“任务树根节点、父子关系、叶子任务、计划阈值、
  实际分配数量、是否超限、是否新建子 AI、子 AI 数量、检查/整合批次数、
  返工次数、质量问题、等待时间、最终判断”。
- 启动新的任务树前，必须先清理上一批子 AI：关闭已经完成、暂停或不再需要的
  子 AI；不再复用旧子 AI。只要工具侧无法关闭、找不到旧 ID，或 UI 仍显示
  运行残留，必须写入 task tracking、pending 和状态看板，不能默认为已销毁或
  已完成。
- 如果工具侧找不到旧 ID 或 UI 仍显示残留，但已确认没有可控活跃子 AI，
  可以按用户批准继续创建全新的子 AI；这些 UI 残留只作为风险记录，不作为
  可复用对象。不得把旧子 AI 换名复用，也不得把残留子 AI 当作新任务承载者。
- 每个用户问题或用户批准后的执行任务，都要视为一个可恢复的“智能体组”：
  主对话保持前台响应和总控协调，后台智能体组按任务树继续处理该问题对应的
  具体工作。前台不得因为后台任务运行而停止回应用户的新问题；后台也不得因为
  前台继续对话而丢失恢复现场。
- 当前会话同时存在多个智能体组时，必须在 task tracking 和必要的 pending 文件中
  维护“智能体组状态看板”，至少记录：智能体组总数、组 ID、组标题、父节点、
  叶子节点数量、运行中/暂停/完成/待验证状态、当前处理文件、未完成项、
  未关闭或无法确认的子 AI、下一步动作、对应 task tracking、pending 恢复文件、
  最近批准标签、最近安全停止点和最小恢复读取清单。用户询问状态时，先展示
  该看板摘要，再继续执行。
- 智能体组看板的结构化事实源是 `.codex/agent-groups/current-status.json`。
  启动、暂停、关闭子 AI，或智能体组状态发生变化时，必须同步更新该文件。
  展示状态时优先运行 `python scripts/agent_group_status.py --once`，默认只显示
  活跃或未完成的智能体组；需要审计历史完成组时再运行
  `python scripts/agent_group_status.py --once --all`。需要持续观察时使用
  `python scripts/agent_group_status.py --watch --interval 5 --max-iterations N`，
  避免留下无限运行的后台进程。脚本无法直接向聊天窗口主动发消息；主对话负责
  调用脚本并把摘要展示给用户。状态文件字段变更或阶段收口后，要运行
  `python scripts/agent_group_status.py --once --validate`，确认手写 summary
  与脚本实时计算一致。
- 子 AI 通信的持久事实源是 `.codex/agent-comm/`。创建新通信组时运行
  `python scripts/agent_comm.py init <group-id> --title "<title>" --write-scope <path>`，
  注册子 AI mailbox 时运行 `python scripts/agent_comm.py register <group-id> <agent-id>`。
  需要确认的消息用 `send --requires-ack`，子 AI 用 `ack` 回执；长任务用
  `heartbeat` 写心跳，用 `artifact` 记录交付物；并发写入用
  `lock acquire/release/status` 管理租约。阶段收口时运行
  `python scripts/agent_comm.py report <group-id>` 和
  `python scripts/agent_comm.py validate <group-id>`，并把结论写入 task tracking。
- 中断、暂停、失败或用户纠正分工规则时，必须立即通知当前子 AI 暂停，
  收集已改文件、未完成项、是否有子任务运行，再关闭或登记无法关闭状态；
  不允许让旧子 AI 在后台继续按过期分工写文件。
- 如果因为上下文强相关、文件强耦合、工具限制或用户明确要求而没有继续拆分，
  必须在 task tracking 说明原因、风险边界和补救验证方式。
- 后续只有在多次任务记录显示阈值过小导致等待成本过高，或阈值过大导致质量下降、
  返工增多、冲突增多时，才提出调整阈值；调整前必须说明数据来源和理由，
  等用户批准后再改 `AGENTS.md` 或相关 skill。
- 本仓库协作机制必须保持自我更新和动态进化：发现新的分工指标、质量指标、
  冲突指标或 commit 节奏问题时，先写入 task tracking 和 corrections，
  再根据频率、严重性和通用性决定是否升级为长期规则或工具化检查。

## 修正文档与规则自迭代

- 每次用户指出 Codex 漏处理、规划不清、文档结构不合理、审批流程遗漏、
  引用维护遗漏、验证不足或违反本仓库要求时，必须把该问题记录到
  `.codex/corrections/`。
- `.codex/corrections/` 采用多文件结构维护：`README.md` 说明用途和模板，
  `index.md` 汇总状态，每次纠错使用一个 `YYYY-MM-DD-错误关键词.md`
  独立记录。不要把所有纠错堆进单个 Markdown 文件。
- 每份纠错记录至少包含：用户纠错摘要、发生场景、错误类型、具体遗漏、
  根因判断、即时修复动作、候选规则、是否需要升级到 `AGENTS.md`、
  关联 task tracking 和当前状态。
- 纠错状态固定为：`待记录`、`待提炼`、`已提炼进要求`、`暂不升级`、
  `已废弃`。更新独立纠错文件状态时，必须同步更新
  `.codex/corrections/index.md`。
- 修正文档只记录 Codex 工作流错误和规则缺口，不记录普通知识点问答，
  不作为正式学习材料入口，不参与 `questions/` 文档层级。
- `.codex/corrections/` 的独立纠错文件是事实源，`index.md` 只是派生汇总。
  如果索引和独立记录冲突，优先相信独立记录，并重建索引。
- 处理 corrections、README、路线、`.references/`、task tracking、索引、
  状态统计等可能被多个会话同时维护的文件前，必须执行“并行会话冲突检查”：
  先重新读取目标文件当前内容，查看目标文件相关 diff 或 `git status --short`，
  必要时运行只读一致性脚本，不能用旧缓存直接覆盖当前文件。
- 并行会话冲突检查要先区分事实源和派生汇总：独立 correction、源码和正式正文
  优先作为事实源，`index.md`、`.references/`、统计表和 task tracking 汇总
  通常是派生或审计材料。发现二者不一致时，可以按事实源重建派生文件，
  但必须在 task tracking 记录发现方式、疑似来源、处理动作、风险边界和验证结果；
  如果冲突涉及同一段落、同一状态统计、同一引用记录或无法判断事实源，
  先在对话中说明冲突并请求用户确认。
- 每次处理 corrections 数据时，都要记录处理动作：读取了哪些记录、
  如何分类或聚类、哪些继续观察、哪些升级规则、是否出现重复人工步骤。
  处理动作至少写入本次 task tracking；涉及索引状态、统计或工具化观察时，
  还要同步写入 `.codex/corrections/index.md`。
- 不能把单次用户纠错原话直接堆进 `AGENTS.md`。必须先在修正文档沉淀，
  再根据频率、严重性和通用性提炼为可复用规则。
- 用户要求“写进要求”“整理修正文档”“根据错误升级规则”或类似任务时，
  必须使用本仓库 `.codex/skills/self-correction-planner/` skill 的流程：
  读取 corrections 索引和未提炼文件，按错误类型聚类，先给规则升级计划，
  等用户批准后再修改 `AGENTS.md`、README 或相关 skill，并回写纠错状态。
- `self-correction-planner` 初期按文字流程执行，不预设固定工具化次数阈值。
  每次使用后要在 `.codex/corrections/index.md` 的工具化观察区域记录流程类型、
  累计次数、最近使用日期、重复步骤、关联记录和当前判断。
- 当工具化观察显示某类流程频繁、重复人工步骤明显或数据处理成本升高时，
  先提出工具化计划并等待用户批准。第一阶段工具只输出统计报告，
  不自动修改 `AGENTS.md`、README、skill 或纠错记录。

- `.source-projects/qt-learning-projects/qt-demo-collection/Qt/`、`.source-projects/qt-learning-projects/qt-demo-collection/Qml/` 这类 Qt5/qmake `.pro` 老示例只作为概念阅读、历史对照或迁移材料，不作为当前默认运行练习入口；除非明确用 Qt6 Kit 验证通过或已经完成 Qt6 迁移。
- `qt-demo-collection/Qt/` 里的网络、JSON、SQLite、线程、绘图、WebSocket、RemoteObjects、QSettings、平台 API 等知识点仍然有学习价值；回答或整理时不要因为它们是 Qt5/qmake 就直接丢弃，而要先提炼知识点，再优先寻找 Qt6 官方示例、本地 Qt6/CMake 示例、`qt-widgets-course/` 兼容示例或已确认 Qt6 的真实项目作为可运行练习入口。
- 回答“哪些 Qt 项目可以学/可以跑”时，要把“可阅读参考”和“本机 Qt6 可运行练习”分开说明，不能默认把本地所有 Qt 快照都当作 Qt6 可运行项目。

## 简历维护规则

本目录下 `resumes` 的简历用于真实投递，修改时必须遵守以下规则。

### 基本定位

- 候选人：贺志豪，本科，湖南文理学院，计算机科学与技术，2025 届。
- 当前定位：刚工作 1 年左右，简历必须做成 1 页投递版。
- 4 个方向分别维护：Java 后端、Go 后端、C++ 开发、Rust/Tauri 客户端。
- 求职意向使用招聘市场常见岗位名：Java 后端开发工程师、Golang 后端开发工程师、C++ 开发工程师、Rust 开发工程师。
- 不要写占位符，不要保留 `XXX`、`<...ROOT>`、学习文件、八股文件、源码路径等内部痕迹。
- 每份简历的“专业技能”第一条必须写“Vibe Coding”，控制在 PDF 中约 1 到 1.5 行；需要体现 Claude Code/Codex、AGENTS.md/SKILL.md、辅助编码、审查测试；不要写成完全依赖 AI 生成代码，不要只罗列工具名，也不要放在基本信息和工作经历之间。

### 页数与版面

- 每份 PDF 必须控制在 1 页。
- 简历内容要尽量写满 1 页，不能只写到半页或留下大面积空白；底部空白不能明显过大，自动检查中 bottom blank 超过约 14% 时要继续调版式或补充真实内容。
- A4 上边距优先使用约 0.75cm；如果 PDF 内容顶部从约 28-32pt 开始，属于正常。
- 顶部空白不能太贴边，自动检查中 top blank 低于约 4% 时必须增加 header 顶部留白；如果视觉上仍偏贴边，可调整到约 5% 左右。调整后必须重新检查底部空白和 1 页结果。
- 如果只需要让某一份简历整体内容下移约 1%，优先微调该简历 Markdown 顶部 `resume-header` 的 `padding-top`，不要直接修改公共 `resume.css` 或 Markdown PDF 全局 margin；调整后必须重新导出该方向 PDF，并同时检查 top blank、bottom blank 和 1 页结果。
- 底部空白优先控制在约 8% 到 12%；如果超过约 12% 且视觉上仍空，应优先补充项目经历里的真实技术细节，而不是拉长自我评价。
- 底部空白不应明显大于顶部空白；如果底部空白过大，优先补充工作经历或项目经历中的真实职责、联调验证和工程能力要点，其次小幅调整行高/段距，不要靠放大字号或拉长自我评价解决。
- 简历版面以最终 PDF 观感为准，不能机械追求每行写满；长段文字、项目条目、工作经历条目、自我评价和项目描述优先控制为 1 行。单个条目换成两行时不机械要求第二行长度，先看整体是否自然；如果同一区域出现多条第二行只多一两个字/词的短尾换行，必须通过压缩、扩写或调整顺序处理，可把第二行超过可用宽度约 40% 作为处理目标。
- 同一区域的条目要有自然错落感，避免多条长度只差一两个字、每条都刚好溢出形成阶梯感，也避免所有句子都撑满整行；参考专业技能区那种长短错落效果。如果当前区域只有一条短尾换行且下方已有自然两行条目，优先把短尾条目放到该区域靠后或最后；两行条目位置服从最终 PDF 效果，不机械全部放最后。
- 工作经历、项目经验和自我评价的句式不能大量重复同一个开头，尤其不要每条都以“参与”开头；在不夸大职责的前提下交替使用“梳理、配合、围绕、通过、结合、补充、完成、定位、验证”等动词，让阅读节奏自然。
- 正文按常见简历规范控制在 10-12pt；Markdown PDF/CSS 中用 `14px` 左右作为正文基准，不要为了填满页面随意放大字体。
- 不要靠大量空格或 `&emsp;` 撑版式；基本信息和工作经历行使用 HTML table 控制列宽。
- 表格不能显示边框，`resumes/resume.css` 中必须保留 `table, tr, td, th { border: none !important; }`。
- 简历正文不要使用 Markdown 反引号代码样式；工具名、文件名直接写普通文本，避免导出后出现不同颜色。

### 照片规则

- `resumes/简历照片.jpg` 必须使用高清原图，不要替换成 72x100 的缩略图。
- 顶部照片使用 `resume-header` 容器内的独立浮动图片，不要放进基本信息表格的 `rowspan` 单元格，否则 100px 照片会把 4 行基本信息撑得过高：
  `<div class="resume-header"><img class="aa" src="./简历照片.jpg"> ... </div>`
- 如果导出的照片模糊，先检查是否误用了压缩小图。

### Markdown PDF 插件规则

- 使用 VS Code 插件 `Markdown PDF` 导出。
- 不要在 Markdown 正文里写 `<style>`，插件 2.1.0 会清洗或转义内联 style 标签，可能导致 CSS 被打印到 PDF。
- 样式统一放在 `resumes/resume.css`。
- 工作区配置 `../.vscode/settings.json` 必须加载该 CSS，并关闭页眉页脚。
- 每次修改简历 Markdown、`resumes/resume.css`、照片、Markdown PDF 配置或导出脚本后，重新导出 PDF 后必须检查重新生成的实际结果，不能只检查文件时间戳。
- PDF 结果检查至少包括：是否 1 页、顶部/底部留白是否失衡、内容是否明显没写满、照片是否清晰且未拉伸、表格是否无边框、正文是否出现代码样式或异常颜色；如果底部还空出明显一截，不能算完成。
- 顶部基本信息区必须清晰可读，姓名、学校、学历、手机、求职意向、专业、毕业时间、邮箱等行不能挤在一起；同时 header 到“工作经历”等首个标题之间不能留下明显大空白。调整版式时要单独检查 header 行距、照片位置和 header 下方间距。
- 如果导出后的 PDF 结果明显变差，例如底部大面积空白、标题/正文过大、内容拥挤、照片模糊或表格出现边框，必须继续调整并重新导出检查，不能把有明显问题的 PDF 当作完成结果。

### 内容取舍

- 正式工作经历写 3 条左右，突出对应方向的真实业务职责。
- 专业技能写 5 条左右，必须匹配对应方向。
- 工作经历和项目经历可以对应同一正式项目，但必须分层表达：工作经历写岗位范围、协作对象、持续职责、交付和问题闭环；项目经历写项目背景、个人角色、技术链路、关键实现、异常处理和验证点。
- 工作经历和项目经历不要复制同一句话；同一事项可出现两次，但工作经历用概括口径，项目经历用细节口径。
- 项目经历写 1 个主项目，必要时通过自我评价补足页面。
- 项目经历条目顺序优先按“核心模块/功能实现 -> 关键场景/异常处理 -> 复现记录/运行排查 -> 验证闭环”收束；“运行排查”“复现记录”“验证闭环”这类总结性条目一般放在项目经历靠后或最后，不要夹在核心功能条目中间打断阅读。
- 如果 PDF 底部空白仍然明显，优先补充项目经历里的真实职责、关键场景、异常处理、联调验证或工程细节；不要主要依赖自我评价撑版面。
- 每份简历底部可以有“自我评价”，用于填满页面；自我评价最多 3 条，每条必须控制为 PDF 中 1 行。
- 自我评价只能作为简短收口，不能在视觉上占据大面积版面；如果自我评价区域下方空白过多，应回到工作经历或项目经历补充真实内容。
- 自我评价要分层覆盖不同侧重点，优先通过具体工作行为隐含“学习能力/技术成长、问题处理/抗压心态、协作沟通/交付意识”三类能力，不要三条都写技术点。
- 自我评价不能使用“学习能力:”“抗压心态:”“协作交付:”这类刻意标签，也不要在句首显式标注能力分类；必须通过具体工作行为体现对应能力，例如梳理源码入口、处理反复异常、记录联调结论和验证结果。
- 自我评价必须具体、真实，优先绑定岗位相关技术、可验证经历和项目行为；可以体现学习能力、抗压心态、协作意识，但必须落到具体场景，不写“认真负责、学习能力强、抗压能力强、沟通能力强”等空泛套话。
- 不要把同一件事在工作经历、项目经历、自我评价里重复三遍；如果页面太空，补充不同层面的真实细节。

### C++ 方向专项写法

- C++ 方向主项目是定制化 Chromium 浏览器内核改造，项目经验优先围绕“指纹配置 -> Renderer JS API -> Mojo IPC -> Browser 进程状态同步 -> 已运行进程广播 / 新 Renderer 配置继承 -> TabHelper 页面埋点 -> 自定义插件加载 -> 中间页检测 / 自动打开 -> GN/Ninja 构建接入 -> 排查验证”组织。
- 如果 Mojo 状态同步链路是本人实现，工作经历、专业技能和项目经验中必须使用“实现 Renderer JS API 到 Browser 进程的配置同步”这类口径，不要弱化成“了解 Mojo”“参与梳理链路”。
- C++ 项目条目不要连续使用“参与、梳理、配合、关注”这类偏弱动词；在不夸大职责的前提下，优先使用“实现、补充、调整、核对、定位、验证”等能体现开发动作的表达。
- 当前 C++ 简历暂不把 WebRTC/代理作为主项目条目；如果需要补一个定制功能，优先从 146-wenmai 的内置中间页选择环境检测、店铺信息获取、自动打开和异常提示等业务功能。
- 插件加载相关条目必须突出“自定义插件加载方式”，可以写从 profile Extensions 目录识别版本和 manifest、ExtensionService 初始化注册、启动后可见性验证，不要只写“插件加载排查”。
- C++ 工程能力可以补 GN/Ninja 编译日志、新增文件和接口接入、构建失败、符号缺失等验证点；Chromium 使用 GN/Ninja，CMake 只能作为一般工程工具了解，不能写成 Chromium 主构建链路。构建接入、异常排查和验证闭环不要拆成三条重复内容，优先合并为“构建接入”和“排查验证”两类，分别讲编译接入问题和运行时复现确认。
- C++ 技能区里 RAII、智能指针、STL、多线程基础属于岗位基础能力，真实掌握时不要只写“了解”；可以写“熟悉 C++ STL、RAII、智能指针、多线程基础和常见数据结构，能结合日志定位运行问题”。
- 实习经历第二条不要泛写“封装通用能力”，优先落到本地文件读写、邮件发送、数据库查询等具体工具接口；实习内容仍保持短，不抢正式工作和主项目篇幅。

- 每次修改任意方向简历的工作经历、专业技能、项目经验或自我评价时，必须提取新增或强化的技术名词、协议、框架、工具、模块和项目链路，检查仓库内是否已有对应的详细知识点文档。

### BOSS 在线简历同步

- 每次修改任一方向正式简历 Markdown 的求职意向、工作经历、专业技能、项目经验、自我评价、项目口径或岗位定位时，必须同步修改对应的 BOSS 在线简历文档；如果对应文档不存在，按 `resumes/BOSS在线简历-方向.md` 新建。
- BOSS 在线简历文档只保留当前可直接填写或复制到平台的最终内容，例如个人信息、个人优势、期望职位、工作经历、项目经历、教育经历和打招呼话术；禁止写“先改这几处”“建议删除/下架”“不要这样写”“面试时口径”“修改说明”“为什么这么改”等过程性或指导性内容。
- BOSS 在线简历文档必须与当前正式简历方向、最新项目口径和岗位定位保持一致；正式简历删除、替换或新增的模块，BOSS 在线简历也要同步删除、替换或新增。
- 修改 BOSS 在线简历文档本身不需要导出 PDF；但只要同时修改正式简历 Markdown、样式、照片或 Markdown PDF 配置，仍必须按 Markdown PDF 插件规则重新导出并检查对应 PDF。
